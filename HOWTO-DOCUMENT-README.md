# Howto Document
The purpose of a README file is to provide the reader with an easy-to-understand and well-structured overview of a topic.
It serves as a central starting point that provides access to further documentations and resources through direct links.

# Basic Guide

## Overall Structure
1. **Headline** </br> Start with a clear and informative title. This should immediately indicate what the document is about. Also possible is to use the folders name as the headline.
2. **Introduction and Description** </br> Introduce the topic and state the 
   - Content 
   - Purpose
3. **Linking** </br> All the files in the folder should be linked in the README file. They can either be presented as a bullet point list or included in the text.
4. **Images** </br> If available, images can be included to better visualize the content.

## Levels
It is important to know for which of the levels a README is written because the structure can slightly differentiate and ech level addresses different kind of readers

- **Level 1: Linking the main topics** - short description of each main topic and a link to the README file for the specific topic (level 2)
- **Level 2: Main topic with links to specialized topics** -see "Overall Structure"
- **Level 3: Specialized topics with links to technical details** - see "Overall Structure" can also include more detailed description and examples 
- **Level 4: Technical details and backgrounds** - see "Overall Structure" but can include more technical details and examples

## Persona Concept

### Jeff
Age: 41 <br/>
Job Title: Project Manager <br/>
Experience: 12 years in project coordination <br/>
Technical Knowledge: Basic understanding of software development

Goals
- Keep the project organized
- Ensure deadlines are met
- Coordinate communication between teams and clients
- Monitor project progress
- Manage risks and priorities

Responsibilities
- Organizing meetings and schedules
- Tracking project milestones
- Communicating with stakeholders
- Managing resources and tasks
- Ensuring smooth teamwork

Needs from the Documentation
- Easy navigation
- Clear summaries
- High-level explanations
- Progress and feature overviews
- Minimal technical complexity

Dokumentation Levels Used

**Level 1**: an overview of the documentation <br/>
**Level 2**: To understand how the different parts of the project work together.  <br/>
**Level 3**: rarely used but can be helpful to look at detailed explanations and examples for specific features.  <br/>
**Level 4**: is not being used
